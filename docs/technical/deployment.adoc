== Umgebungsübersicht

Das System wird in mehreren Umgebungen betrieben, die unterschiedliche Zwecke haben.
Jede Umgebung ist vollständig isoliert. Eine Änderung in einer Umgebung beeinflusst
keine andere.

[cols="1,1,2,1,1", options="header"]
|===
| Umgebung | URL | Zweck | Branch | Deployment

| Lokal
| http://localhost:8080
| Entwicklung und manuelle Tests auf dem eigenen Rechner
| Beliebig
| Manuell (Docker Compose)

| CI
| —
| Automatische Qualitätsprüfung bei jedem Push
| Alle Branches
| Automatisch (GitHub Actions)

| Staging
| [z. B. https://staging.example.com]
| Integrationstest; Abnahme durch Product Owner vor Release
| `main`
| Automatisch nach Merge in `main`

| Produktion
| [z. B. https://example.com]
| Live-System für Endnutzer:innen
| `main` (getaggt)
| Manuell mit Freigabe
|===

NOTE: In der Hochschulumgebung entfällt oft eine separate Produktionsumgebung.
In diesem Fall dient Staging gleichzeitig als Produktionsumgebung.

=== System lokal starten

[source,bash]
----
# 1. Repository klonen
git clone [Repository-URL]
cd [projektname]

# 2. Umgebungsvariablen konfigurieren
cp .env.example .env
# .env mit einem Editor öffnen und ggf. anpassen

# 3. Alle Dienste starten
docker compose up --build

# 4. Prüfen, ob alles läuft
# Frontend:  http://localhost:3000
# Backend:   http://localhost:8080
# API-Docs:  http://localhost:8080/swagger-ui
----

=== System stoppen und aufräumen

[source,bash]
----
# Dienste stoppen (Daten bleiben erhalten)
docker compose down

# Dienste stoppen und Volumes löschen (Datenbank wird geleert)
docker compose down -v
----

== CI/CD-Pipeline

Die Pipeline automatisiert Qualitätsprüfung und Deployment. Sie stellt sicher,
dass kein Code in `main` gelangt, der Tests bricht oder Qualitätsstandards verletzt.

=== Pipeline-Schritte im Detail

[cols="1,2,1,1", options="header"]
|===
| Schritt | Beschreibung | Schlägt fehl bei | Ausgeführt bei

| Linting
| Code-Stil und statische Analyse prüfen
| Stilfehler oder Warnungen über Schwellwert
| Jedem Push

| Unit-Tests
| Alle Unit-Tests ausführen
| Fehlschlagende Tests
| Jedem Push

| Integrationstests
| Tests gegen echte Datenbank (Testcontainers)
| Fehlschlagende Tests
| Jedem Push

| Abdeckungsprüfung
| Testabdeckung messen und mit Mindestwert vergleichen
| Unterschreitung des Mindestwerts (siehe Testkonzept)
| Jedem Push

| Image-Build
| Docker-Images für alle Services bauen
| Build-Fehler
| Jedem Push auf `main` und PRs

| Staging-Deployment
| Neue Images auf Staging-Umgebung ausrollen
| Deployment-Fehler oder fehlschlagende Smoke Tests
| Merge in `main`

| Produktions-Deployment
| Neues Release auf Produktion ausrollen
| Deployment-Fehler oder fehlschlagende Smoke Tests
| Manuell nach Tag-Erstellung
|===

=== Pipeline-Konfiguration

Die vollständige Pipeline-Konfiguration befindet sich in:

[source]
----
.github/workflows/
├── ci.yml          ← Qualitätsprüfung (bei jedem Push)
└── deploy.yml      ← Deployment (Staging & Produktion)
----


== Deployment-Ablauf

=== Deployment auf Staging (automatisch)

Jeder Merge in `main` löst automatisch ein Deployment auf Staging aus.
Kein manueller Eingriff erforderlich.

. Pipeline baut neue Docker Images
. Images werden in die Container-Registry gepusht
. Staging-Umgebung zieht die neuen Images
. Smoke Tests werden automatisch ausgeführt
. Team wird bei Erfolg oder Fehler benachrichtigt

=== Release und Deployment auf Produktion (manuell)

[source,bash]
----
# 1. Sicherstellen, dass main aktuell und stabil ist
git checkout main
git pull

# 2. Release-Tag erstellen (Semantic Versioning)
git tag v1.2.0
git push --tags

# 3. In GitHub Actions: Produktions-Deployment manuell freigeben
# → GitHub → Actions → deploy.yml → "Run workflow" → Version bestätigen
----

NOTE: Wir verwenden _Semantic Versioning_ mit dem Schema MAJOR.MINOR.PATCH: 
MAJOR bei inkompatiblen Änderungen,
MINOR bei neuen Features (abwärtskompatibel),
PATCH bei Bugfixes.

| Login-Flow

== Rollback

Ein Rollback ist notwendig, wenn ein Deployment kritische Fehler in der
Produktionsumgebung verursacht. Schnelles Handeln minimiert die Auswirkungen.

=== Wann wird ein Rollback durchgeführt?

* Kritische Fehler in Produktion nach einem Deployment
* Smoke Tests schlagen nach Deployment fehl
* Fehlerrate oder Antwortzeiten steigen signifikant an

=== Rollback-Ablauf

. *Entscheidung treffen* — Tech Lead oder Product Owner entscheiden über Rollback
. *Stakeholder informieren* — Team und relevante Stakeholder benachrichtigen
. *Rollback ausführen* (eine von zwei Optionen):

[source,bash]
----
# Option A: Letztes stabiles Tag erneut deployen
# → GitHub → Actions → deploy.yml → Version des letzten stabilen Tags eingeben

# Option B: Manuell das vorherige Image aktivieren
docker compose pull [service]:[vorherige-version]
docker compose up -d [service]
----

[start=4]
. *Smoke Tests ausführen* — sicherstellen dass das System wieder stabil ist
. *Ursache analysieren* — was hat das Problem verursacht?
. *Nachbereitung dokumentieren* — im Ticket oder als Kommentar im ADR

[cols="1,3", options="header"]
|===
| Aspekt | Beschreibung

| Log-Format
| Strukturiertes JSON-Logging; enthält Zeitstempel, Level, Service, Message

| Log-Zugriff (lokal)
