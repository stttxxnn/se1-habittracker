== Testarten

=== Unit-Tests

Unit-Tests prüfen einzelne Klassen oder Funktionen isoliert voneinander.

[cols="1,3", options="header"]
|===
| Aspekt | Beschreibung

| Ziel
| Sicherstellung der korrekten Funktionsweise einzelner Komponenten in Isolation

| Umfang
| Alle Service- und Domänenklassen; Fokus auf Geschäftslogik

| Werkzeug
| [z. B. JUnit 5, Pytest, Jest]

| Mocking (Faken / Simulieren)
| [z. B. Mockito, unittest.mock — externe Abhängigkeiten werden gemockt]

| Abdeckungsziel
| [z. B. mind. 80 % Zeilenabdeckung auf der Service-Schicht]

| Ausführung
| Lokal bei jedem Build; automatisch in der CI-Pipeline bei jedem Push
|===

=== Integrationstests

Integrationstests prüfen das Zusammenspiel mehrerer Komponenten, z. B. Service und Datenbank.

[cols="1,3", options="header"]
|===
| Aspekt | Beschreibung

| Ziel
| Sicherstellung der korrekten Kommunikation zwischen Komponenten und mit externen Systemen

| Umfang
| [z. B. REST-Endpunkte, Datenbankzugriffe, externe API-Aufrufe]

| Werkzeug
| [z. B. Spring Boot Test, Testcontainers, Supertest]

| Testdaten
| [z. B. dedizierte Testdatenbank, In-Memory-Datenbank, Fixtures]

| Ausführung
| Automatisch in der CI-Pipeline; nicht zwingend bei jedem lokalen Build
|===

=== Optional: End-to-End-Tests (E2E)

E2E-Tests simulieren echte Nutzerinteraktionen durch die gesamte Anwendung.

//E2E-Tests sind optional und sollten erst betrachtet werden, wenn alle anderen Testarten ausreichend berücksichtigt wurden.

[cols="1,3", options="header"]
|===
| Aspekt | Beschreibung

| Ziel
| Sicherstellung der wichtigsten Interaktionen aus Endnutzerperspektive

| Umfang
| [z. B. Login, Kernfunktion A, Kernfunktion B — nur kritische Pfade]

| Werkzeug
| [z. B. Cypress, Playwright, Selenium]

| Ausführung
| [z. B. manuell vor jedem Release; oder automatisch auf Staging-Umgebung]
|===

=== Manuelle Tests

Hier werden alle Tests aufgeführt, die nicht automatisiert werden können.

[cols="1,3", options="header"]
|===
| Aspekt | Beschreibung

| Ziel
| Überprüfung von Usability, visueller Korrektheit und explorativem Testen

| Umfang
| [z. B. UI-Design, Responsivität, neue Features vor dem Sprint Review]

| Vorgehen
| [z. B. Exploratives Testen durch Teammitglied, das das Feature nicht implementiert hat]

| Dokumentation
| [z. B. Testergebnisse als Kommentar in GitIssue]
|===

== Testabdeckung

_Definiert die angestrebten Mindestwerte für die Testabdeckung._

[cols="1,1,2", options="header"]
|===
| Schicht | Zielwert | Bemerkung

| Service-Schicht
| [z. B. ≥ 80 %]
| Geschäftslogik hat höchste Priorität

| Persistenz-Schicht
| [z. B. ≥ 60 %]
| Grundlegende CRUD-Operationen abgedeckt

| Controller / API
| [z. B. ≥ 70 %]
| Wichtigste Endpunkte per Integrationstest abgedeckt

| Frontend-Komponenten
| [z. B. ≥ 50 %]
| Fokus auf wiederverwendbare Kernkomponenten
|===

NOTE: Abdeckung ist kein Selbstzweck. 80 % sinnlose Tests sind schlechter als 60 % gut durchdachte Tests. Qualität der Tests ist wichtiger als der Prozentwert.

== Integration in den Entwicklungsprozess

=== Ausführung der Tests
==== Lokale Entwicklung
[source,bash]
----
# Alle Unit-Tests ausführen
[z. B.  pytest  |  npm test]

# Testabdeckung prüfen
[z. B.  pytest --cov]
----

==== CI-Pipeline

Tests werden automatisch bei jedem Push im Repository ausgeführt. Je nach Testart können unterschiedliche Trigger und Umgebungen sinnvoll sein. Die Konfiguration der CI-Pipeline sollte sicherstellen, dass alle Tests zuverlässig und schnell ausgeführt werden.

=== Definition of Done

//Bitte anpassen und nur das aufnehmen, was tatsächlich auch durchgeführt wird.
Folgende Testkriterien sind Teil unserer Definition of Done:

* [ ] Alle bestehenden Tests sind grün
* [ ] Neue Funktionalität ist durch Unit-Tests abgedeckt
* [ ] Die Mindest-Testabdeckung wird nicht unterschritten
* [ ] Manuelle Tests für neue Features wurden durchgeführt
* [ ] Keine bekannten kritischen Fehler sind offen
